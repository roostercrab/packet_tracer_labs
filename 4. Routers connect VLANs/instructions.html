Routing between the VLANs

The switches are configured to have two separate VLANs that are connected through a trunk. Obviously having one computer per VLAN per switch is a little bit wasteful, but that’s all we need in Packet Tracer to prove basic functionality.

We are going to configure this in two ways, the first way is going to be incorrect but will demonstrate why we do it the (correct) way that we do. Specifically, in the first example we will use one physical interface on the router for each VLAN. This is not necessarily an unreasonable solution, because on a router with enough ports that needed full bandwidth of that link it might even be an acceptable solution. This router has only two ports and they are fake computers on it so it does not meet that criteria, nor will most networks that are VLAN’d.

But most times when a network is VLAN’d the individual VLANs only require access to the router and they can share that interface’s bandwidth with the other VLANs that will be on the same link. In fact the router is usually only used to provide access to the Internet as most devices inside a network never need to talk to each other; even devices on the *same* VLAN don’t need to talk to each other, but we will look at that later on with Private VLANs.

So just for an example we are going to configure the router to use each of its two interfaces to connect the two VLANs

 

VLAN routing method 1 (incorrect):

1.      Open up routers_connect_vlans_wrong_way_begin.pkt

2.      On Switch0 create a link between int fa0/23 and fa0/0 on Router0

3.      On Switch0 create a link between int fa0/24 and fa0/1 on Router0

4.      We have to make the links off Switch0 into access interfaces that are members of the VLAN that will have the default gateway for that network on Router0; the other side of the link.

 

Make the two links on the switches into access interfaces for the VLAN that they will be associated with using the following commands:

 

en

conf t

int fa0/23

switchport mode access

switchport access vlan 12

int fa0/24

switchport mode access

switchport access vlan 13

 

5.      On Router0 give each of the interfaces the IP address that is associated with the default gateway for that subnet. So since interface fa0/0 is now associated with vlan 12 and we told the computers on vlan 12 that their default gateway was 192.168.12.254; and since interface fa0/1 is associated with vlan 13 and we told the computers there that their default gateway was 192.168.13.254; then we need to configure int fa0/0 on Router0 to be 192.168.12.254 and int fa0/1 to be 192.168.13.254:

 

en

conf t

int fa0/0

ip address 192.168.12.254 255.255.255.0

no shutdown

int fa0/1

ip address 192.168.13.254 255.255.255.0

no shutdown

 

6.      Now if you try to ping from a computer on the vlan 12 subnet to a computer on the vlan 13 subnet you will be able to get through. This is because the sending computer will recognize that the destination IP address is not on its same subnet and so it will ARP for the MAC address of its default gateway. It will then forward the packet to the MAC address of the router which because interface fa0/23 is associated with vlan 12 it will reach interface fa0/0 on the router where it will receive it, remove the layer 2 information, check its routing table for a network with that destination address in it and then forward it out that interface if it finds a matching entry.

7.      You can verify all these things with these commands (remember if you are in config mode to add a “do” in front of these show commands):

On Router0 look at the routing table:

show ip route

On Switch0 look at the CAM table (mac-address-table):

              show mac-address-table

!!! PROTIP !!! you can see what devices have which MAC address in Packet Tracer by hovering over them

 

On the computers you can see that they have resolved their default gateway to the MAC address of the interface that has their default gateway on it (Desktop>Command Prompt):

              arp -a

8.      Open routers_connect_vlans_wrong_way_completed.pkt if you would like to see the completed network.

VLAN routing method 2 (correct):

1.      Open up routers_connect_vlans_right_way_begin.pkt

2.      The router has had the wires removed, but I want you to take off the IP addresses that we had just had configured on there with these commands:

 

en

conf t

int fa0/0

no ip address

int fa0/1

no ip address

 

3.      Connect a wire between fa0/0 on Router0 and fa0/24 on Switch0

4.      Now take the access vlans off the switch (we’re just going to set them back to what they were and change fa0/24 to a trunk so that it can carry both vlans over the same line):

 

en

conf t

int fa0/23

no switchport access vlan

int fa0/24

no switchport access vlan

switchport mode trunk

 

5.      Now that the switch will treat fa0/24 as a member of all vlans (because it is a trunk) we can configure subinterfaces on Router0’s fa0/0:

 

en

conf t

int fa0/0.12

!!! That command creates subinterface 12 on fa0/0, it will act like a unique physical interface

encapsulation dot1q 12

!!! This command will tell the router that if it sees anything come in tagged with vlan 12 (dot1q is Cisco’s cute shorthand for 802.1q, the vlan protocol) that it will direct it to this “interface” as though it were a physical interface.

ip address 192.168.12.254 255.255.255.0

!!! Notice that this is the same command that we originally put onto fa0/0

6.      Do the same thing to create a subinterface for vlan 13:

 

int fa0/0.13

encapsulation dot1q 13

ip address 192.168.13.254 255.255.255.0

 

7.      Whenever you create subinterfaces, you take away the ability for the main interface to have a usable IP address, but you have to make sure that the main interface is not shutdown or else the subinterfaces will be shutdown along with it:

 

int fa0/0

no shutdown

 

8.      Now you should be able to ping from one computer network to the other one! Run these same commands to see that the router has both networks in its routing table, that the switch has learned of the MAC addresses out its interfaces, and that the computers have associated the MAC address of their default gateway with the IP address of their default gateway.

Last question as food for thought, because this is what will make you a good diagnostician:

 

Does it matter that the computers all use the same MAC address for their default gateway?

 

There is also a 3rd way that we will explore shortly, giving a switch a small “router brain” so that it can make these decisions instead of forwarding them to the router just to have them come back down the same wire with a different vlan tag.
